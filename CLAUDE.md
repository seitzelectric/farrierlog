## Project overview

FarrierLog is an offline-first Flutter business app for farriers/field-service professionals to manage clients, horses, visits, invoices, and photos. SQLite (via `sqflite`) is the only persistence layer — there is no backend or cloud sync. Primary target is Android (`applicationId com.seitzelectric.farrierlog`), but the standard Flutter desktop/web/iOS scaffolding is also present.

## Commands

```bash
flutter pub get                 # install dependencies
flutter analyze                 # static analysis (flutter_lints rules in analysis_options.yaml)
flutter test                    # run all tests
flutter test test/widget_test.dart            # run a single test file
flutter test --plain-name "<test name>"       # run a single test by name
flutter run                     # run the app (select device when prompted)
flutter build apk               # build Android release APK
```

Tests that touch `DatabaseService` initialize `sqflite_common_ffi` and point `databaseFactory`/`databasesPath` at a temp directory (see `test/widget_test.dart`) instead of using the real on-device sqflite plugin. Always call `DatabaseService.close()` in `tearDown` so the static `_db` singleton doesn't leak between tests.

## Architecture

### Layering

- `lib/models/models.dart` — plain data classes (`Client`, `Horse`, `Visit`, `ServiceLine`, `VisitPhoto`, `InvoiceRecord`, and join helpers like `HorseWithClientInfo`). Every model has `toMap()`/`fromMap()` for SQLite rows; there is no ORM.
- `lib/services/` — all business logic and I/O, as static-method classes (no DI, no instances):
  - `database_service.dart` — schema, migrations, and all CRUD/query methods. This is the single source of truth for SQL.
  - `backup_service.dart` — full backup/restore (DB + media files) to a zip archive, with manifest validation.
  - `export_service.dart` — CSV export (clients/animals/visits/service lines/invoice summary) zipped together.
  - `invoice_service.dart` — PDF invoice generation/printing/sharing, plus persisted `CompanyInfo` (the farrier's own business details used on invoices).
- `lib/screens/` — one `StatefulWidget` per screen, each loading its own data directly from `DatabaseService` in `initState`/`_loadData` (no global state management — no Provider/Bloc/Riverpod). Screens commonly re-run `_loadData()` after returning from a pushed route or dialog to pick up changes.
- `lib/widgets/widgets.dart` and `lib/screens/screens.dart` are barrel files that `export` the individual files in their directory — import these barrels rather than individual files when adding new screens/widgets that need access to sibling types.
- `lib/utils/utils.dart` — `AppUtils` static helpers: date/time/currency formatting and Google Maps URI building.

### Database

- Single sqflite database, versioned with `onCreate`/`onUpgrade` in `database_service.dart`. Current version is tracked by `_dbVersion`; every schema change must add an `oldVersion < N` branch in `_onUpgrade` AND bump `_dbVersion` — there are no separate migration files.
- Foreign keys are enforced (`PRAGMA foreign_keys = ON`); cascading deletes are used extensively (deleting a client cascades to horses, visits, service lines, photos).
- Tables: `clients`, `horses`, `visits`, `visit_horses` (join table), `service_lines`, `visit_photos`, `invoices`, `app_settings` (key/value store, created in `onOpen` so it always exists regardless of version).
- Recurring visits: a `Visit` has `recurrenceWeeks` and `next_recurring_visit_id`. Confirming an auto-generated visit (`confirmAutoGeneratedVisit` in `database_service.dart`) creates/links the next projected visit in the chain rather than eagerly generating all future visits.

### Photos and files

Photos, invoices, and backups are stored as files on disk (paths recorded in the DB, e.g. `VisitPhoto.path`, `InvoiceRecord.filePath`), under app-managed directories via `path_provider`. `BackupService` copies these media files into the backup zip and rewrites paths on restore — when adding new file-backed data, make sure it's included in both backup and restore paths.

### Versioning

Android `versionCode` in `android/app/build.gradle` is hardcoded (not driven by `flutter.versionCode` from `local.properties`) — bump it manually for each release build, separately from the `version:` field in `pubspec.yaml`.

## Database Migration Pattern

FarrierLog uses sqflite's built-in `onCreate`/`onUpgrade` pattern — no separate migration files.

When adding a schema change:
1. Add an `if (oldVersion < N)` block in `_onUpgrade` in `database_service.dart`
2. Bump `_dbVersion` to `N`
3. Update `onCreate` if the new column/table should exist on fresh installs

Next migration: **9**

## Photo and File Storage

Photos, invoices, and backups are stored as files on disk. Paths are recorded in the DB (`VisitPhoto.path`, `InvoiceRecord.filePath`). App-managed directories via `path_provider`.

Photos are stored under `getApplicationDocumentsDirectory()/photos/` with UUID-based filenames.

When adding any new file-backed data type, include it in both `BackupService` backup and restore paths.
